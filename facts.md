# [1] The Impact of SBOM Generators on Vulnerability Assessment in Python: A Comparison and a Novel Approach
Authors: Giacomo Benedetti, Serena Cofano, Alessandro Brighente & Mauro Conti 
## 1. Key Quantitative Results

This process allows us to have a sample with the same proportion of package managers identified in the previous steps, and generalize our results to the entire package ecosystem with a 3.04% margin of error at 95% confidence level by standard sample size calculations"*

    > *"We manually scraped the list of tools on the CycloneDX tool center. We obtained a list of 169 open-source tools... We reduce the list to 24 elements... Eventually, we obtained a list of 5 tools: cdxgen, GH-sbom, ORT, Syft, and Trivy"*

    > *Table 3 data: "cdxgen | 978/5" and "PIP-sbom | 47/3" under F. Poss./F. Negs. column; also: "reduces by ten times the number of false positives"*

## 2. Qualitative Conclusions and Interpretations

*   **Conclusion:** This study represents the first security analysis evaluating how SBOM generation accuracy impacts vulnerability detection capabilities
    > *"In this paper, we perform the first security analysis on the vulnerability detection capabilities of tools receiving SBOMs as input"*  
    > 📍 *Location: Abstract, para. 2*

*   **Conclusion:** Current state-of-the-art SBOM generation tools are unsuitable for proper vulnerability assessment of Python projects
    > *"Takeaway: Current state-of-the-art SBOM generation tools are not suitable to generate SBOMs for the proper vulnerability assessment of Python projects"*  
    > 📍 *Location: Section 6.1 RQ1.1, final paragraph*

*   **Conclusion:** Two root causes identified for poor SBOM tool performance: inadequate package manager support and failure to model actual dependency collection behavior
    > *"Takeaway: (1) SBOM generation tools do not properly support multiple package managers. (2) SBOM generation tools do not consider how dependencies are actually collected by Python's package managers"*  
    > 📍 *Location: Section 6.1 RQ1.2, final paragraph*

*   **Conclusion:** cdxgen's superior performance attributed to virtual environment installation and broad package manager support
    > *"Among the analyzed SBOM generation tools, the one providing the best vulnerability scan results is cdxgen. This is motivated by the fact that cdxgen (1) installs dependencies in a virtual environment, and (2) supports most of the package managers"*  
    > 📍 *Location: Section 6.1 RQ1.1, para. 2*

*   **Conclusion:** Native integration of SBOM generation into package managers significantly improves vulnerability assessment accuracy
    > *"using native resources of the ecosystem—e.g., the package manager—to generate the SBOM may largely improve the overall security posture of software"*  
    > 📍 *Location: Section 10 Conclusion and Future Work, para. 1*

*   **Conclusion:** High false positive rates from baseline tools create investigation burden that may discourage SBOM adoption
    > *"While an overestimation is generally considered better than losing vulnerabilities [2, 4, 14], these numbers are out-of-scale, causing burdening during vulnerability investigation"*  
    > 📍 *Location: Section 6.1 RQ1.2, para. 3*

## 3. Methodological Details for Reproducibility

*   **Parameter:** Ground truth vulnerability set constructed via six-step automated pipeline using pip-audit
    > *"For each package in our collected sample we automatically: (1) retrieve the package's project from its code repository; (2) parse metadata files to obtain optional dependency groups; (3) install the package in a virtual environment along with both required and optional dependencies; (4) generate the requirements.txt file using the pip freeze command to filter out packages installed in the virtual environment by default; (5) pass the requirements.txt to pip-audit; (6) collect the security report"*  
    > 📍 *Location: Section 3.3 Security Report Ground Truth, para. 1*

*   **Parameter:** Jaccard similarity index used to quantify overlap between tool-identified vulnerabilities and ground truth
    > *"J(ToolVulns, GrTrVulns) = |ToolVulns ∩ GrTrVulns| / |ToolVulns ∪ GrTrVulns|"*  
    > 📍 *Location: Section 5 RQ1.1, Equation (1)*

*   **Parameter:** Precision and recall metrics calculated per standard definitions to evaluate vulnerability detection performance
    > *"Precision = TP / (TP + FP)"* and *"Recall = TP / (TP + FN)"*  
    > 📍 *Location: Section 5 RQ1.2, Equations (2) and (3)*

*   **Parameter:** PIP-sbom architecture implements dual-process design: online PyPI interaction for dependency resolution, offline graph construction for SBOM generation
    > *"PIP-sbom includes an online process and an offline process. The online process interacts with the PyPI registry obtaining the dependency network. The offline process builds the dependency network graph and generates the SBOM document from a Python project"*  
    > 📍 *Location: Section 4 PIP-SBOM: Our Proposed SBOM Generator, para. 4*

*   **Parameter:** Dependency network modeled as directed unweighted graph G=(V,E) where edges represent transitive dependency relationships
    > *"The dependency network is defined as a direct unweighted graph G = (V, E). Each element n ∈ V is either a direct or transitive dependency of the input project r ∈ V. An edge (u, v) ∈ E represents a dependency relationship between the nodes u and v"*  
    > 📍 *Location: Section 4, Dependency Network Graph subsection*

*   **Parameter:** SBOM output format complies with CycloneDX standard, including bom-ref, name, version, purl in components field and dependency relationships in dependencies field
    > *"An entry contains: the bom-ref, dependency name, version and purl... Edges of the dependency network are used to fill the dependencies field of the SBOM, which contains the relationship between components... PIP-sbom produces a CycloneDx-compliant SBOM"*  
    > 📍 *Location: Section 4, SBOM Generation subsection*

## 4. Limitations and Future Directions

*   **Limitation:** Sample generalizability constrained by potential omission of edge cases despite statistical validity
    > *"Our sample can be generalized to the whole package population on PyPI, with the margin of error stated in Sect. 3.1. While it is relevant because of generalization, it may miss specific corner cases providing insightful knowledge"*  
    > 📍 *Location: Section 8 Threats to Validity, Projects Collection subsection*

*   **Limitation:** Tool selection process involved manual filtering introducing potential subjectivity and exclusion of viable alternatives
    > *"The selection is manually conducted by filtering SBOM generation tools based on the criteria discussed in Sect. 3.2... However, a degree of subjectivity would be present in the selection, leading to the exclusion of other potentially effective tools"*  
    > 📍 *Location: Section 8 Threats to Validity, SBOM Generation Tools Selection subsection*

*   **Limitation:** Ground truth reliance on pip-audit may introduce undetected vulnerabilities affecting baseline comparisons
    > *"The reliance on pip-audit for establishing the ground truth introduces potential limitations, as this tool may not detect all relevant vulnerabilities, which could impact the baseline used for comparing other tools"*  
    > 📍 *Location: Section 8 Threats to Validity, Ground Truth subsection*

*   **Limitation:** Findings specific to Python ecosystem; transferability to other languages not empirically validated
    > *"The methodology and results are tailored to Python, while we consider these results extensible to other languages, we cannot state their transferability"*  
    > 📍 *Location: Section 8 Threats to Validity, Evaluation Scope subsection*

*   **Limitation:** Vulnerability identifier mismatches across databases may contribute to measured false positives/negatives
    > *"We experienced false positives and negatives while evaluating vulnerability scan results. Some are due to a missing vulnerability identifier inside the vulnerability scan result. These errors can be easily fixed by establishing a unique vulnerability database mapping vulnerabilities to their identifiers on the various databases"*  
    > 📍 *Location: Section 8 Threats to Validity, Evaluation Methodology subsection*

*   **Future Direction:** Extend native SBOM generation approach to additional software ecosystems beyond Python
    > *"Some future works in this direction can support the generation of SBOM in different software ecosystems"*  
    > 📍 *Location: Section 10 Conclusion and Future Work, para. 2*

# [2] A Comparative Study of Vulnerability Reporting by Software Composition Analysis Tools
Authors: Nasif Imtiaz, Seaver Thorn, Laurie Williams

## 1. Key Quantitative Results

*   **Fact:** The study compared 9 industry-leading SCA tools on OpenMRS, a web application composed of 43 Maven projects and 5 npm projects.
    > *"OpenMRS consists of 44 projects that are hosted in their own separate repositories on GitHub. Out of the 44 projects, 39 are Maven projects and 1 is a npm project. The other 4 projects are composed of a Maven and a npm project each."*  
    > 📍 *Location: Section 3: Evaluation Case Study: OpenMRS, para. 1*

*   **Fact:** OpenMRS contains 547 unique Maven dependencies (311 unique packages) and 2,213 unique npm dependencies (1,498 unique packages).
    > *"Total unique dependencies (package and version): Maven: 547, npm: 2,213 | Total unique packages: Maven: 311, npm: 1,498"*  
    > 📍 *Location: Table 1: OpenMRS dependency overview*

*   **Fact:** The count of reported vulnerable dependencies varied widely: 17–332 for Maven projects and 32–239 for npm projects across studied tools.
    > *"The count of reported vulnerable dependencies ranges from 17 to 332 for Maven and from 32 to 239 for npm projects across the studied tools. Similarly, the count of unique known vulnerabilities reported by the tools ranges from 36 to 313 for Maven and from 45 to 234 for npm projects."*  
    > 📍 *Location: Abstract, Results paragraph*

*   **Fact:** OWASP Dependency-Check reported the highest number of unique vulnerable dependencies for both ecosystems: 332 for Maven and 239 for npm.
    > *"OWASP DC detects the highest number of unique dependencies and unique vulnerabilities for both Maven and npm projects."*  
    > 📍 *Location: Section 5.1: RQ1 Findings, para. 2*

*   **Fact:** Commercial B, which monitors dependencies during runtime, reported the lowest count of vulnerable Maven dependencies: 35 unique dependencies and 127 unique vulnerabilities.
    > *"Commercial B that monitors the dependency under use during runtime detected the lowest number of vulnerable dependencies for Maven projects."*  
    > 📍 *Location: Section 5.1: RQ1 Findings, para. 2*

*   **Fact:** Only 2 of 5 npm-scanning tools (npm audit and Snyk) report vulnerable dependency paths, which is critical because npm can introduce the same package through multiple transitive paths.
    > *"We find that only npm audit and Snyk report all possible dependency paths to each unique vulnerability."*  
    > 📍 *Location: Section 5.1: RQ1 Findings, para. 4*

*   **Fact:** Reported vulnerabilities are mostly introduced through transitive dependencies, except for Dependabot and WhiteSource which show higher rates of direct dependency vulnerabilities.
    > *"We find that reported vulnerabilities are mostly introduced through transitive dependencies, except for Dependabot and WhiteSource. The latter two tools assist GitHub projects in automatically fixing the vulnerable dependencies (by upgrading to a safer version), which may explain the high rate of direct dependencies in their reporting."*  
    > 📍 *Location: Section 5.1: RQ1 Findings, para. 6*

*   **Fact:** OWASP DC detected 42 JavaScript dependencies in Maven projects that were not declared in Maven dependency files; WhiteSource detected 20 such dependencies.
    > *"In total, 42 JavaScript dependencies were found by OWASP DC, while WhiteSource found 20."*  
    > 📍 *Location: Section 5.2: Why do the tools differ?, reason (1)*

*   **Fact:** Steady's static analysis found only 2.1% of alerts had potentially executable vulnerable code; dynamic analysis found 1.6% actually executed.
    > *"Steady found only 2.1% of the alerts were potentially executable and 1.6% of the alerts were actually executed."*  
    > 📍 *Location: Section 5.3.1: Code analysis-based metrics, para. 3*

*   **Fact:** Of 53 non-CVEs reported by Snyk, 41 were published before 2020; of 54 non-CVEs reported by WhiteSource, 50 were published before 2020.
    > *"Of the 53 non-CVEs reported by Snyk, 41 were published before 2020; while of the 54 non-CVEs reported by WhiteSource, 50 were published before 2020."*  
    > 📍 *Location: Section 6: Discussion, para. 3*

## 2. Qualitative Conclusions and Interpretations

*   **Conclusion:** Accuracy of the vulnerability database is identified as a key differentiator among SCA tools.
    > *"Our manual analysis of the tools' results suggest that accuracy of the vulnerability database is a key differentiator for SCA tools."*  
    > 📍 *Location: Abstract, Results paragraph*

*   **Conclusion:** Practitioners should not rely on any single SCA tool, as this can result in missing known vulnerabilities.
    > *"We recommend that practitioners should not rely on any single tool at the present, as that can result in missing known vulnerabilities."*  
    > 📍 *Location: Abstract, Conclusion paragraph*

*   **Conclusion:** Inconsistency in mapping vulnerabilities to affected package versions may result in inaccurate alerts and tool divergence.
    > *"Incorrect mapping of a vulnerability to the affected version range of a package may result in inaccurate alerts."*  
    > 📍 *Location: Section 5.2: Why do the tools differ?, reason (4)*

*   **Conclusion:** Non-CVEs reported by different tools cannot be automatically cross-referenced due to lack of common identifiers, complicating validation and comparison.
    > *"However, we have no reliable way to map non-CVEs across different tool reports."*  
    > 📍 *Location: Section 4.3: Analyzing Tool Results, para. 2*

*   **Conclusion:** Tools that offer automated fix suggestions should also explain potential backward incompatibility risks, as fear of breaking changes is a primary reason developers avoid updating dependencies.
    > *"However, prior work has found that fear of breaking change is one of the primary reasons developers do not want to update vulnerable dependencies [21]. Tools may provide a more in-depth analysis on what code change is there with a certain version change in a dependency and if there is any possibilities of introduction of regression bugs."*  
    > 📍 *Location: Section 6: Discussion, para. 5*

*   **Conclusion:** Developers may need to evaluate dependency vulnerability risks on a case-by-case basis, as not all reported vulnerabilities are relevant to the client application.
    > *"Prior research has shown that not all vulnerabilities in dependency may be relevant for the client application [38, 39]. [...] The lack of a common framework for risk assessment among SCA tools suggest that developers will have to evaluate the vulnerability alerts case-by-case basis based on their expertise on the project codebase and how the dependencies are being used by the specific project."*  
    > 📍 *Location: Section 6: Discussion, para. 6*

## 3. Methodological Details for Reproducibility

*   **Parameter:** Tool selection criteria required that tools (a) scan either Maven or npm projects, (b) be accessible as an executable, and (c) offer unique features compared to already-selected tools.
    > *"From the relevant search results, we filtered the tools with the following inclusion criteria: a) scans either Maven or npm projects; b) we have access to an executable tool; and c) offers unique features when compared with already selected tools."*  
    > 📍 *Location: Section 4.1: Selection Criteria, para. 1*

*   **Parameter:** All tools were executed during September 2020 to ensure fair comparison of vulnerability data, except Steady whose database was from January 2020.
    > *"As vulnerability data gets updated over time, we ran all the tools during September 2020 to ensure a fair comparison, except for Steady whose vulnerability data is from January 2020."*  
    > 📍 *Location: Section 4.2: Tool Description, final paragraph*

*   **Parameter:** For manual analysis of tool differences, the first author inspected results from the "coreapps" project (largest dependency count), and the second author independently verified the categorization.
    > *"We specifically focused on the project coreapps as this is the project with the largest dependency count and includes both Maven and npm dependencies. The first author went through results from all the tools for coreapps, and categorized the differences. The second author then independently went through the results from the studied tools and verified the categorization done by the first author."*  
    > 📍 *Location: Section 4.3: Analyzing Tool Results, para. 5*

*   **Parameter:** Steady was configured in a virtual machine with 16 GB RAM and 4 processor cores; reachability analysis included static call graph construction, JUnit test execution tracing, and JVM instrumentation (though the third analysis could not complete).
    > *"We set up Steady in a virtual machine, allocating 16 GB RAM, and 4 processor cores. [...] For reachability analysis of the identified vulnerabilities, Steady performs three analyses: 1) static call graph construction; 2) executing JUnit tests for analyzing executability traces; and 3) JVM instrumentation through integration testing. We were unable to complete the third analysis as the tool presumably ran out of memory after running for ten days."*  
    > 📍 *Location: Section 4.2: Tool Description, Eclipse Steady paragraph*

*   **Parameter:** Commercial B's runtime analysis used OpenMRS's 123 integration test cases interacting via Selenium webdriver to monitor dependency usage.
    > *"To run this tool on OpenMRS, we make use of 123 test cases provided by OpenMRS for integration testing that interact with the application through a Selenium webdriver."*  
    > 📍 *Location: Section 4.2: Tool Description, Commercial B paragraph*

## 4. Limitations and Future Directions

*   **Limitation:** Single-case-study design limits generalizability to other package ecosystems (e.g., Ruby, Rust) or applications with different dependency management practices.
    > *"We evaluate 9 SCA tools on one web application at a certain release point, which poses a threat to the generalizability of our findings. [...] However, our findings may not generalize to other package ecosystems, such as Ruby or Rust. Similarly, any other application with a difference in dependency management may have yield different findings."*  
    > 📍 *Location: Section 7: Limitations, para. 1*

*   **Limitation:** Absence of ground truth for SCA alerts makes it nontrivial to determine (a) all open source dependencies in use, (b) correctness of reported vulnerability data, or (c) exploitability of dependency vulnerabilities.
    > *"Another limitation of our study involves the absence of ground truth. In the context of SCA alerts, there can be three steps for building such a ground truth - a) determining all the open source dependencies in use; b) determining the correctness of the reported vulnerability data; c) determining exploitability of the dependency vulnerabilities. These steps are nontrivial to perform manually for a real software and may be more suitable with a synthetic test subject [28]."*  
    > 📍 *Location: Section 7: Limitations, para. 2*

*   **Limitation:** Exploitability assessment was out of scope because only Commercial B claims to filter dependencies based on actual usage; other tools do not make such claims.
    > *"Regarding exploitability, no tools except Commercial B claims to filter out dependencies on any use criteria, while Steady and Commercial A only offers additional analysis to aid in such contextual assessment. Therefore, exploitability would not be a fair comparison criteria for the studied SCA tools, and out-of-scope for this study."*  
    > 📍 *Location: Section 7: Limitations, para. 2*

*   **Limitation:** No statistical significance testing was conducted for RQ1 due to the single-case-study design.
    > *"Further, for RQ1, we do not conduct any statistical test to measure significance in difference, as we only perform a single case study."*  
    > 📍 *Location: Section 7: Limitations, para. 2*

*   **Future Direction:** Establish frameworks and metrics to identify false positives for dependency vulnerabilities.
    > *"We point out two research directions in the SCA space: i) establishing frameworks and metrics to identify false positives for dependency vulnerabilities"*  
    > 📍 *Location: Abstract, Conclusion paragraph*

*   **Future Direction:** Build automation technologies for continuous monitoring of vulnerability data from open source package ecosystems.
    > *"and ii) building automation technologies for continuous monitoring of vulnerability data from open source package ecosystems."*  
    > 📍 *Location: Abstract, Conclusion paragraph*

*   **Future Direction:** Research is needed to understand how reliably all open source components (including code fragments) used in a deployed application can be identified.
    > *"Future research is necessary to understand how reliably we can identify all the open source components used in a deployed application."*  
    > 📍 *Location: Section 6: Discussion, para. 1*

*   **Future Direction:** Evaluate whether developers in practice find code-analysis-based risk metrics (e.g., reachability analysis) useful for prioritizing vulnerability fixes.
    > *"However, future research is required to evaluate if developers in the real world actually use such metrics and find them useful or not."*  
    > 📍 *Location: Section 6: Discussion, para. 6*

# [3] Accuracy Evaluation of SBOM Tools for Web Applications and System-Level Software
Authors: Andreas Haigitler, Dominik Merli (Technical University of Applied Sciences Augsburg, Germany)

## 1. Key Quantitative Results

*   **Fact:** Four programming languages were selected for comparative analysis: two for web applications (Python, Typescript) and two for system-level software (C, Rust).
    > *"we present a selection of four popular programming languages: Python, C, Rust and Typescript. For web application software we consider Python and Typescript while for system-level software C and Rust are investigated."*  
    > 📍 *Location: Abstract*

*   **Fact:** Twelve evaluation criteria were applied: seven NTIA minimum elements, four recommended elements, and one completeness criterion.
    > *"The NTIA defines the sections Supplier Name (SN), Component Name (CN), Version of the Component (VOC), Other Unique Identifiers (OUI), Dependency Relationship (DR), Author of the SBOM Data (AOSD) and Timestamp (TS)"* and *"Furthermore the NTIA defines a recommended section for an SBOM. It includes: Hash of the component (HOC), Lifecycle phase (LP), Other component relationship (OCR) and License information (LI)."*  
    > 📍 *Location: Section 3.2-3.3*

*   **Fact:** Tool completeness results varied significantly: SBOM4Python failed completeness for Python; Syft and Cdxgen failed for C; Cdxgen failed for Rust; all tools achieved completeness for Typescript except where noted.
    > *Table 1 data: Python/SBOM4Python (C: -); C/Syft (C: -), C/Cdxgen (C: -); Rust/Cdxgen (C: -); Typescript: all tools (C: +)*  
    > 📍 *Location: Table 1, Section 5*

*   **Fact:** Dependency relationship accuracy was achieved by only specific tool/language combinations: Cdxgen (Python), CycloneDX-Conan-Extension (C), SBOM4Rust (Rust), and Cdxgen/CycloneDX-Npm (Typescript).
    > *"The dependency relationship is only depicted correctly by Cdxgen in Python, by CycloneDX-Conan-Extension in C, by SBOM4Rust in Rust and by Cdxgen and CycloneDX-Npm in Typescript."*  
    > 📍 *Location: Section 6, Discussion, para. 2*

*   **Fact:** Best-performing tools by language: Cdxgen for Python (7/12 criteria), CycloneDX-Conan-Extension for C (8/12), no clear best for Rust (all 6/12), CycloneDX-Npm for Typescript (9/12).
    > *"The best tool for Python is Cdxgen as it provides seven of twelve data fields... The best tool for C is CycloneDX-Conan-Extension it provides eight of twelve data fields... There is no best tool for Rust as every of the three tools provide six of twelve data fields... The best tool for Typescript is CycloneDX-Npm. It provides nine of twelve data fields"*  
    > 📍 *Location: Section 6, Discussion, paras. 4-7*

*   **Fact:** Specific tool versions were used to ensure reproducibility: Syft 0.90.0, Cdxgen 9.7.5, CycloneDX-Python 4.1.1, SBOM4Python 0.10.3, CycloneDX-Npm 1.14.3, Covenant 0.16.0.
    > *"Syft, version 0.90.0... Cdxgen, version 9.7.5... CycloneDX-Python, version 4.1.1... SBOM4Python, version 0.10.3... CycloneDX-Npm, version 1.14.3... Covenant, version 0.16.0"*  
    > 📍 *Location: Section 4.3.1-4.3.5*

## 3. Methodological Details for Reproducibility

*   **Parameter:** Sample projects were intentionally minimized to enable manual verification of SBOM output.
    > *"For manual checking, the software projects are kept small with a small amount of packages and a single dependency."*  
    > 📍 *Location: Abstract*

*   **Parameter:** Tool selection was restricted to official CycloneDX-listed, open-source tools to ensure transparency and quality.
    > *"For ensuring quality only official tools of the CycloneDX website are part of the investigation. As transparency is important for securing the software supply chain only open-source tools are part of the inspection."*  
    > 📍 *Location: Section 4.2*

*   **Parameter:** Tools were categorized as either general-purpose or programming language-dependent to enable comparative analysis.
    > *"Additionally, we categorize the tools in general and programming language dependent tools, because not every single tool can work with every programming language."*  
    > 📍 *Location: Section 4.2*

*   **Parameter:** CycloneDX in JSON format was selected over XML or SPDX due to superior human readability for manual criterion verification.
    > *"As the focus of the paper is checking the NTIA criteria by hand, human readability is important. JSON is easier for readability and therefore chosen."*  
    > 📍 *Location: Section 4.1*

*   **Parameter:** Package selection prioritized zero-dependency packages, with exactly one single-dependency package per language to test dependency mapping.
    > *"Only packages with zero dependencies were chosen, except one package with one dependency."*  
    > 📍 *Location: Section 4.4*

*   **Parameter:** All tools were executed in standard/default configuration without additional arguments to establish baseline performance.
    > *"The tools were run in their standard configuration."*  
    > 📍 *Location: Section 4.5*

*   **Parameter:** Evaluation scoring used three symbols: "+" for fully present criteria, "-" for absent, "o" for partially present or incorrectly formatted; only "+" and "-" counted toward accuracy metrics.
    > *"If the criteria is present in the SBOM then there is a '+', if the criteria is not present then there is a '-'. [...] As the requirement focuses on accuracy of evaluation only '+' and '-' are counted."*  
    > 📍 *Location: Section 5, para. 1*

## 4. Limitations and Future Directions

*   **Limitation:** Many evaluated tools fail to produce complete SBOMs, omitting test packages and dependencies.
    > *"Many tools do not provide a complete SBOM, as they do not depict every test package and dependency."*  
    > 📍 *Location: Abstract*

*   **Limitation:** Study scope excluded proprietary tools; performance of closed-source SBOM generators remains unknown.
    > *"Part of the current investigation were only open-source tools. It can not be determined whether proprietary tools may perform better or worse in generating SBOMs."*  
    > 📍 *Location: Section 6, Discussion, para. 10*

*   **Limitation:** Tools were tested only in default configuration; optimized parameter settings might yield improved results.
    > *"As the tools were started in their standard configuration there may be the possibility that certain tools may produce better results if they are started with more arguments."*  
    > 📍 *Location: Section 6, Discussion, para. 11*

*   **Limitation:** CycloneDX standard version inconsistency (1.4 vs. 1.5) across tools may affect comparability of outputs.
    > *"Also the versions of the CycloneDX standards vary. Some tools provide an CycloneDX standard in version 1.4 and some tools provide the newer standard in version 1.5."*  
    > 📍 *Location: Section 6, Discussion, para. 9*

*   **Limitation:** Manual verification becomes infeasible for large-scale SBOMs, potentially masking completeness errors in production use.
    > *"Especially the fact that packages or dependencies are missing in some tools require at least an additional check for example from a human, which can not be assured when the SBOM is bigger than a human readable size."*  
    > 📍 *Location: Section 7, Conclusion, para. 2*

*   **Parameter:** Tool selection criteria required GitHub projects labeled 'sbom' with ≥100 stars, CycloneDX format output, JavaScript/npm support, and CLI functionality.

A custom Python script then processes this tree, extracting vital details: component names, their versions, and interdependencies."*

*   **Parameter:** Evaluation metrics were defined as: Accuracy = TP/(TP+FP+FN), Precision = TP/(TP+FP), Recall = TP/(TP+FN), with explicit acknowledgment that true negatives do not apply in this evaluation context.

*   **Parameter:** Reproducibility was assessed by generating SBOMs twice for each project using all four tools, excluding serialNumber and timestamp fields which are expected to vary.

*   **Parameter:** Dependency verification employed a two-step process: (a) mapping components to dependencies from SBOM outputs and npm list output, and (b) scoring matches against ground truth.

In parallel, the command npm list yields a hierarchized dependency structure. We parse this to form a comparable mapping that links each package (with its version) to direct dependencies. *   **Parameter:** Definition of a program construct for unique identification.

    > *"To obtain the distance between abstract syntax trees we use tree differencing algorithms (Falleri et al. 2014; Fluri et al. 2007). ## 2. Qualitative Conclusions and Interpretations

*   **Conclusion:** Dependency networks grow over time in both size and update frequency, but a minority of packages accounts for most updates.
    > *"We observe that the dependency networks tend to grow over time, both in size and in number of package updates, while a minority of packages are responsible for most of the package updates."*  
    > 📍 *Location: Abstract*

*   **Conclusion:** Most packages depend on others, but only a small proportion accounts for most reverse dependencies.
    > *"The majority of packages depend on other packages, but only a small proportion of packages accounts for most of the reverse dependencies."*  
    > 📍 *Location: Abstract*

*   **Conclusion:** A high and increasing proportion of transitive dependencies creates "fragile" packages vulnerable to failure propagation.
    > *"We observe a high proportion of 'fragile' packages due to a high and increasing number of transitive dependencies."*  
    > 📍 *Location: Abstract*

*   **Conclusion:** The proposed Changeability, Reusability, and P-Impact Indices enable cross-ecosystem comparison despite differences in age and size.
    > *"Another novelty is that we introduce three new metrics to facilitate ecosystem comparison despite the diversity of the considered ecosystems is terms of age and size: the Changeability Index captures an ecosystem's propensity to change over time; the Reusability Index captures the ecosystem's amplitude and extent of reuse; and the P-Impact Index assesses the fragility of an ecosystem."*  
    > 📍 *Location: Section 1, para. 5*

*   **Conclusion:** CRAN's "rolling release" policy likely explains its lower Changeability Index, lower transitive-to-direct dependency ratio, and shallower dependency trees.
    > *"This is very likely due to the fact that CRAN imposes a stricter policy on its package maintainers than the other considered package managers."* / *"CRAN follows a 'rolling release' policy that imposes packages to be up-to-date with their dependencies"*  
    > 📍 *Location: Section 8.1, para. 1–2*

*   **Conclusion:** The higher Reusability Index for npm may be explained by JavaScript's intentionally small standard library.
    > *"The higher values and growth rate for npm could be explained by the relative poorness of JavaScript's standard library."*  
    > 📍 *Location: Section 6, para. 9*

*   **Conclusion:** The findings provide initial empirical evidence for Lehman's laws of software evolution (Continuing Growth, Continuing Change, Increasing Complexity) at the ecosystem level.
    > *"These three laws focus on structural and technical aspects of software. [...] we can claim to have found initial empirical evidence for the law of Continuing Growth at ecosystem level"* / *"We also found partial empirical evidence for the law of Continuing Change at ecosystem level"* / *"We also found partial support for the law of Increasing Complexity"*  
    > 📍 *Location: Section 10.1, para. 2–4*

*   **Conclusion:** Ecosystem policies significantly influence observed dependency network characteristics.
    > *"These differences highlight ecosystem particularities such as specific policies (e.g., the 'rolling release' policy of CRAN induces a lower Changeability Index)"*  
    > 📍 *Location: Section 11, para. 5*

## 3. Methodological Details for Reproducibility

*   **Parameter:** Data source and licensing.
    > *"For our empirical study, we relied on information about package releases and dependencies collected by the open source discovery service libraries.io (Nesbitt and Nickolls 2017). The extracted data falls under the CC-BY-SA 4.0 license."*  
    > 📍 *Location: Section 3.4, para. 1*

*   **Parameter:** Ecosystem selection criteria: excluded ecosystems with <5,000 packages, domain-specific targets, subsets of already-considered ecosystems, or incomplete dependency metadata.
    > *"We excluded those that we considered too small (less than 5,000 packages). From the remaining 17 packaging ecosystems, we selected seven for which it was possible to obtain all the necessary metadata from the package manifests statically"*  
    > 📍 *Location: Section 3.4, para. 2–3*

*   **Parameter:** Dependency types included: "runtime", "imports", "depends", "normal"; excluded: "development", "optional", "enhances", "suggests", "build", "configure", "test", "develop", "dev".
    > *"Depending on the ecosystem, this means that we restricted ourselves to dependencies of type 'runtime', 'imports', 'depends' and 'normal', while omitting dependencies of type 'development', 'optional', 'enhances', 'suggests', 'build', 'configure', 'test', 'develop' or 'dev'."*  
    > 📍 *Location: Section 3.4, para. 6*

*   **Parameter:** For CRAN, archived package metadata was supplemented using the extractoR tool, adding 1,078 packages and 5,182 releases.
    > *"For CRAN, we completed the metadata extracted from libraries.io with data about archived package releases [...] With the help of extractoR, we retrieved the metadata of 1,078 additional packages and 5,182 additional package releases."*  
    > 📍 *Location: Section 3.4, para. 5*

*   **Parameter:** For npm, ~250 packages designed solely to depend on many other packages (e.g., npm-gen-all) were excluded from analysis.
    > *"We identified around 250 of such packages, and explicitly ignored them for our analyzes since they only introduce noise and do not serve any useful purpose."*  
    > 📍 *Location: Section 3.4, para. 5*

*   **Parameter:** Definition of Changeability Index.
    > *"Definition 1 The Changeability Index of an ecosystem E at time t is the maximal value n such that there exist n packages in E at time t having been updated at least n times during the last month."*  
    > 📍 *Location: Section 5, Definition 1*

*   **Parameter:** Definition of Reusability Index.
    > *"Definition 2 The Reusability Index of an ecosystem E at time t is the maximal value n such that there exist n required packages in E at time t having at least n dependent packages."*  
    > 📍 *Location: Section 6, Definition 2*

*   **Parameter:** Definition of P-Impact Index.
    > *"Definition 3 The P-Impact Index of an ecosystem E at time t is the number of packages in E at time t that are transitively required by at least P% of all the packages in E."*

## 4. Limitations and Future Directions

*   **Limitation:** Data completeness: package removal policies may have excluded packages removed before April 2017 if historical data was not preserved.
    > *"Depending on the ecosystem's package removal policy, packages that were removed from the ecosystem before that date may have been absent from our analysis if no historical data was preserved by the ecosystem after package removal."*  
    > 📍 *Location: Section 9, para. 2*

*   **Limitation:** Vendored and dynamically defined dependencies were not captured, as analysis relied only on explicitly declared manifest dependencies.
    > *"We constructed the dependency networks by relying on the list of dependencies explicitly provided in each package manifest. As a consequence, vendored dependencies and dynamically defined dependencies were not considered in our analyses."*  
    > 📍 *Location: Section 9, para. 3*

*   **Limitation:** Chronological ordering of releases may not match logical versioning order for packages with parallel maintenance branches.
    > *"While this chronological order should match the logical order induced by the versioning scheme in most cases, this is not the case for instance for packages having multiple branches that are maintained in parallel."*  
    > 📍 *Location: Section 9, para. 4*

*   **Limitation:** Some analyses may be affected by local phenomena or data quality issues (e.g., incorrect creation dates for bulk-imported RubyGems releases).
    > *"Some analyses may be affected by local phenomena [...] For instance, the peak in the number of updates in August 2014 for RubyGems (Fig. 3) corresponds to the massive import of 25 K package releases in RubyGems, resulting in an incorrect creation date for those package releases"*  
    > 📍 *Location: Section 9, para. 5*

*   **Limitation:** Findings may not generalize to non-programming-language package managers (e.g., WordPress, Eclipse, Atom) where packages target end-users rather than reuse.
    > *"We do not make any claims that our results can be generalized beyond package dependency networks similar to those that we have analyzed, i.e., the main package managers for specific programming languages."*  
    > 📍 *Location: Section 9, para. 6*

*   **Future Direction:** Automated extraction of dynamic dependencies (e.g., for Maven, PyPI) where dependencies are defined at installation time.
    > *"An interesting topic of future work would therefore constitute the automated analysis and extraction of such dynamic dependencies."*  
    > 📍 *Location: Section 3.4, para. 3*

*   **Future Direction:** Study software ecosystems from a complex networks perspective to identify scale-freeness, small-world phenomena, or power-law behavior.
    > *"Inspired by Cataldo et al. (2014), we hypothesize that package dependency networks of open source packaging ecosystems also reveal such complex network behavior."*  
    > 📍 *Location: Section 10.2, para. 2*

*   **Future Direction:** Extend technical dependency analysis with socio-technical analysis incorporating contributor social networks.
    > *"it would be very useful to study the ecosystem dynamics from a socio-technical point of view, combining information from the package dependency network with information from the social network of ecosystem contributors."*  
    > 📍 *Location: Section 10.3, para. 2*
